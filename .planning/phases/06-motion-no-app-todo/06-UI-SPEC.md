# Phase 6 — Motion Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This phase adds NO new layout, color, typography, or copy. It is a **motion-only progressive-enhancement contract** over the existing `radar-goiania.html` identity. Where the standard UI-SPEC template asks for design tokens, this document confirms the EXISTING tokens are unchanged and specifies motion parameters instead (durations, easings, transforms, triggers, reduced-motion behavior).

No new typography. Stagger/transition animations never change font-size, weight, or line-height mid-animation (that would cause reflow/repaint, defeating the compositor-thread performance goal). Only `transform` and `opacity` are animated.

## Motion Contract

### 1. Screen Transitions (home ⇄ busca ⇄ detalhe via `setView()`)

**Trigger:** every call to `setView(v)` — currently a synchronous `data-view` attribute swap with no animation (hard cut via `display:none`/`display:flex` CSS rules keyed on `body[data-view]`).

**What animates:** the OUTGOING view fades+slides out while the INCOMING view fades+slides in. Only mobile (`≤820px`, single-view-at-a-time via `data-view`) needs this — desktop already shows the map full-bleed with `.panel` as a `position:fixed` overlay, so desktop uses the SAME transform/opacity treatment on `.panel` show/hide (the overlay entrance/exit), not a full-screen swap.

| Property | Value |
|----------|-------|
| Transform | `translateX(12px)` → `translateX(0)` outgoing-to-incoming direction follows the tab order (busca is left of mapa in the `.viewbar`: mapa→busca slides from right, busca→mapa slides from left) — i.e. `±12px`, never more (subtle, not a slide-panel effect) |
| Opacity | `0 → 1` (incoming), `1 → 0` (outgoing) |
| Duration | 180ms |
| Easing | `cubic-bezier(0.22, 1, 0.36, 1)` (Motion's built-in "easeOut"-style curve — decelerate into place, no bounce) |
| Desktop `.panel` overlay | same values, applied to the `.panel` element's own show/hide (replacing the raw `display:flex` toggle at `body[data-view="busca"] .panel{display:flex}` with a Motion `animate()` call gated by `REDUCE`) |
| Interruptible | YES — mandatory. A new `setView()` call while a transition is in flight must cancel the running Motion animation (`animation.stop()` / re-calling `animate()` on the same element restarts from current computed value, Motion does this natively) and start the new one immediately. Never queue, never await completion before allowing the next `setView()`. |
| Leaflet coordination | `setView("mapa")` still calls `map.invalidateSize()` + `refreshLots()` via the existing `setTimeout(...,60)` — do NOT change this timing to wait on the Motion transition; the map remeasure is independent of the visual fade and must not be gated behind animation completion (a slow/interrupted transition must never delay the map becoming interactive) |

**Out of scope:** Leaflet's own `map.fitBounds()`/`map.setView()` camera moves (drillBairro, mostrarBairro, pick) are NOT touched by this phase — no `flyTo` easing added. They remain instant `fitBounds` jumps as today. Motion governs DOM/CSS transitions only, never the Leaflet camera.

### 2. Bottom-Sheet Spring (`.detail` open/close)

**Trigger:** `showDetail()` adds `.show` (today: `d.classList.add("show")`, instant per existing CSS `.detail{display:none} .detail.show{display:block}`); `closeDetail()` removes it.

**What animates:** `translateY(100%) → translateY(0)` on open (mobile only — `.detail` is already `position:absolute;bottom:0` per breakpoint; desktop's `.detail` is a smaller inline card in the `.panel`, not a sheet — spring applies to the MOBILE sheet specifically, `≤820px`).

| Property | Value |
|----------|-------|
| Transform | `translateY(100%) → translateY(0)` (open), reverse (close) |
| Spring params | stiffness `420`, damping `38`, mass `1` (Motion spring type) — tuned for a fast, minimally-overshooting settle (~1 small overshoot bounce max, not a visibly bouncy sheet). If approximated in CSS as fallback: `cubic-bezier(0.34, 1.28, 0.64, 1)` ease over 220ms is the closest single-curve stand-in, but Motion's native spring is preferred since it's already the chosen lib |
| Approx. settle time | ~220-260ms to visually at-rest (springs don't have a fixed "duration" — this is the perceptual target, tune stiffness/damping to land here) |
| Close duration | Same spring, or a slightly snappier non-spring `160ms cubic-bezier(0.4,0,1,1)` ease-in on close (sheets dismissing don't need overshoot — closing fast reads better than closing bouncy) |
| Drag handle coordination | The EXISTING drag-to-dismiss logic (`pointerdown/pointermove/pointerup` on `#grab`, lines setting `d.style.transform` directly during drag) MUST take priority over the Motion spring — while the user's finger is down (`y0!=null`), Motion must not be animating the same `transform` property. On `pointerup`, if `dy<=70` (not dismissed), the existing code resets `d.style.transform=""` — Motion's open-state spring should already have the sheet at `translateY(0)` so this reset is a no-op visually; if `dy>70`, the existing `closeDetail()` call takes over and this phase's spring plays the close animation from the CURRENT dragged position, not from `translateY(100%)` (interruptible mid-drag→animated-close handoff) |
| Must not block interaction | The sheet's internal content (comparables button, action links, back-to-list button) must remain clickable throughout the opening spring — do not use a JS `await` on animation completion before wiring up content; content is already in the DOM and interactive the instant `showDetail()` runs, motion is purely the entrance visual |
| Interruptible | YES — rapid open→close→open (e.g. tapping between two cards quickly) must not queue animations; each new open/close call restarts from the sheet's current transform, never from a stale target |

**Desktop `.detail` card (inside `.panel`):** no spring — desktop's detail view is a smaller card appended within the scrollable panel, not a sheet; apply the SAME fade+small-translateY treatment as screen transitions (translateY 8px + opacity, 180ms) rather than a spring, since there's no "sheet coming up from off-screen" affordance on desktop.

### 3. Stagger — Result Cards + Lot Polygons (FIRST render only)

**Trigger A (cards):** `render(list)` builds the `#results` innerHTML in one synchronous pass today. Stagger applies to the `.card` elements' entrance ONLY on the render call that follows a NEW search (`buscar()`/`loadCi()`/`onMapClick` producing a fresh `list`), never on `render(LAST)` re-render (the "Mostrar mais" pagination button) and never on any re-render of the same list.

**Trigger B (lot polygons):** `refreshLots()` clears and repopulates `lotLayer` on every map pan/zoom that crosses into zoom≥17. Stagger applies ONLY the FIRST time lots appear for a given drill-in (i.e., the transition from "no lots visible" to "lots populated" — e.g., right after `drillBairro()`/zooming past the zoom-17 threshold), NOT on every subsequent pan within the same zoomed-in session (`refreshLots` already short-circuits repeat calls via `LOTLAST.contains(map.getBounds())`, but even on genuine new-bounds fetches after the first reveal, do not re-stagger — anti-feature per FEATURES.md: "stagger only on first render", repeated staggering on pan reads as annoying, not premium).

| Property | Value |
|----------|-------|
| Per-item transform | `translateY(8px) → translateY(0)` + `opacity 0 → 1` |
| Per-item duration | 200ms |
| Per-item easing | `cubic-bezier(0.22, 1, 0.36, 1)` (same decelerate curve as screen transitions — visual consistency) |
| Inter-item stagger delay | 24ms (within the 20-30ms range specified) |
| Max staggered items | cap the stagger at the first ~12 items; items beyond that (long lists, `RENDN=300` cap) animate in with the SAME timing as item 12 (all at once) rather than making the user wait through a multi-second cascade for item #250 — this matches the anti-feature guidance against long/cinematic sequences |
| Lot polygons stagger equivalent | Since Leaflet polygons aren't DOM elements with CSS transitions in the same way, approximate via `fillOpacity`/`opacity` animated with Motion's `animate()` driving a JS callback that calls `poly.setStyle({opacity, fillOpacity})` per frame, OR simpler: stagger the `poly.addTo(lotLayer)` calls themselves with `stagger()`-computed delays (24ms apart, same 12-item cap) so polygons visually "pop in" in sequence rather than a synchronous `fillOpacity 0→target` per polygon if per-polygon animation proves too costly on low-end Android — prefer the simplest implementation that reads as a stagger without adding per-frame JS style-recalc across hundreds of polygons |
| Detection of "first render"/"first reveal" | Cards: a boolean/flag set false at the start of `buscar()`/`loadCi()`/search entry points and consulted once in `render()`, OR simply: stagger always applies in `render()` EXCEPT when called with `RENDN` pagination (`render(LAST)` from the "Mostrar mais" button) — gate on a parameter, e.g. `render(list, {stagger:true})` from search entry points, `render(LAST)` (default `stagger:false`) from pagination. Lots: gate on the LOTLAST transition from `null`/empty→populated (the existing `!map.hasLayer(lotLayer)` check right before `lotLayer.addTo(map)` is already exactly this transition point) |

### 4. Tap Feedback (`:active` states)

**Implementation:** CSS-only, as anticipated in CONTEXT.md ("tap feedback provavelmente CSS"). No Motion/JS involvement — `:active` pseudo-class is synchronous with the browser's own touch/pointer handling, has zero risk of jank, and needs no reduced-motion gate (a scale/darken on active press is not considered "motion" that vestibular-sensitive users need suppressed — it's instantaneous state feedback, not an animated transition; kept regardless of `prefers-reduced-motion`, matching the existing `.go:active{transform:translateY(1px)}` and `.wnext:active{transform:translateY(1px)}` precedents already shipped un-gated).

| Element | Property | Value |
|---------|----------|-------|
| `.card:active` (result card) | `transform: scale(.98)` + slight darken | `transform:scale(.98)`; darken via `filter:brightness(.97)` or a 4-6% mix toward `--ink` on `background` — reuse the existing `.card.sel{background:var(--paper-2)}` mechanism's spirit, don't introduce a new color token |
| `.searchpill:active` | already has `border-color:var(--accent)` on `:active` — ADD `transform:scale(.98)` alongside it (extend, don't replace) |
| `.detail .acts a:active` (action links/buttons in sheet) | `transform:scale(.98)` | new — this class currently only has `:hover` |
| `.viewbar button:active` | `transform:scale(.97)` | new — bottom tab bar, ≥44px targets already, scale is purely visual and keeps the full hit area (touch target size is determined by layout box, not the visual `transform`, which does not shrink the actual tappable region) |
| `.moderow button:active` | `transform:scale(.98)` | new |
| Transition | `transition:transform .1s ease` (CSS, not Motion) so the press-and-release itself feels instant (~100ms), well under the 150-250ms budget reserved for the JS-driven Motion animations above |

**Why CSS not Motion:** `:active` needs to be indistinguishable from input latency — any JS dispatch (even Motion's WAAPI-backed calls) adds a task-queue hop that CSS pseudo-classes skip entirely. This is the correct tool split per STACK.md ("CSS is the correct tool for simple state transitions and costs nothing to load").

### 5. Reduced-Motion — Global Rule + JS Guard

**Present since the FIRST commit of any Motion code in this phase — not a retrofit.**

**CSS (global kill-switch, covers anything that slips through as a CSS transition/animation):**

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.001ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.001ms !important;
    scroll-behavior: auto !important;
  }
}
```

This block does NOT need to (and should not) disable the `:active` scale/darken from Section 4 — the media query above targets `animation`/`transition` durations, and the tap-feedback `transition:transform .1s ease` will still be near-instant-perceived at 0.001ms, which is the desired end state anyway (state change without perceptible animation). No special-casing needed.

**JS guard:**

```js
const REDUCE = matchMedia("(prefers-reduced-motion: reduce)").matches;
```

Declared once, near the top of the app's script (alongside other top-level state), consulted before every Motion `animate()` call:

- **Screen transitions:** if `REDUCE`, `setView()` swaps `data-view`/toggles `.panel` display with NO Motion call at all — the existing instant CSS `display` swap remains exactly as it is today (zero code path change beyond "skip the animate() call").
- **Bottom sheet:** if `REDUCE`, `.detail.show` toggling stays the current instant `display:none/block` — no spring, no translateY animation. Drag-to-dismiss gesture still works unchanged (it's pointer-driven, not Motion-driven, so it's unaffected either way).
- **Stagger:** if `REDUCE`, all cards/polygons appear simultaneously at full opacity with no transform offset — i.e., skip straight to the end state of the animation instantly, which is visually identical to today's un-animated render.
- **Tap feedback:** unaffected (see Section 4 rationale) — always active regardless of `REDUCE`.

**Must also react to LIVE changes**, not just page-load state (a user can toggle the OS setting while the app is open):

```js
matchMedia("(prefers-reduced-motion: reduce)").addEventListener("change", e => { REDUCE = e.matches; });
```

(`REDUCE` becomes a `let`, not `const`, to support this — correct the guard above accordingly at implementation time.)

### 6. Interruptibility + Progressive Enhancement — Degradation Matrix

| Scenario | Required behavior |
|----------|---------------------|
| Motion (window.Motion) fails to be defined (inline script blocked/corrupted paste) | Every call site MUST be wrapped so the ABSENCE of `Motion` never throws and never blocks the underlying state change. Pattern: `if (window.Motion && !REDUCE) { Motion.animate(...) } ` — the state change itself (`data-view` swap, `.show` class toggle, DOM insertion) happens unconditionally OUTSIDE that guard, so app function is 100% independent of Motion loading. |
| `prefers-reduced-motion: reduce` active | See Section 5 — instant state changes, no transforms, no springs, no stagger delays. Tap feedback (CSS `:active`) unaffected. |
| Mid-interaction: new `setView()` fires while a transition is animating | Cancel/restart, never queue. Motion's `animate()` called again on the same element/property automatically takes over from the current animated value (WAAPI behavior) — do not manually track "is an animation running" state to block re-entry; let Motion handle interruption natively. |
| Mid-interaction: sheet is being dragged (`y0!=null`) when some other event tries to open/close it | Existing drag logic already has full control of `d.style.transform` during drag via inline style (higher specificity than any Motion-driven class/attribute animation) — Motion must not fight the drag; only engage on `pointerup`/`pointercancel`/programmatic `closeDetail()`/`showDetail()` calls, never while `y0` is non-null. |
| Mid-interaction: rapid card taps / rapid drill-ins | Screen-transition and stagger animations are per-render, short (150-250ms budget), and cancel-and-restart cleanly — a corretor doing rapid-fire lookups in the field must never feel like they're "waiting for an animation to finish" before the next tap registers. No animation may set `pointer-events:none` on interactive content for its duration. |
| Low-end Android / throttled CPU | All animated properties are `transform`/`opacity` only (compositor-thread, GPU-accelerated) — never `top`/`left`/`width`/`height`/`margin` — so frame budget stays cheap even under load. This is non-negotiable: any implementation animating layout-triggering properties violates this contract. |
| Lot-polygon stagger under many features (near the 4000-record `resultRecordCount` cap in `refreshLots`) | The 12-item stagger cap (Section 3) already bounds worst-case animated-item count; polygons beyond the cap render at full opacity immediately, no perceptible "still loading" wait. |
