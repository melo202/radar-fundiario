# Phase 2 — UI Design Contract: Home = Mapa

> Visual and interaction contract for the map-first home restructuring. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Scope: MAPA-01 (home=mapa, zoom em Goiânia), MAPA-04 (busca vira card/pill). Extends the existing `radar-goiania.html` design system — introduces ZERO new colors, fonts, or component patterns. Everything below is a composition of variables and idioms that already ship in v1.0.

## Component Contracts

### 1. Search Pill

**Purpose:** Always-visible, one-tap affordance that opens the existing full search panel. Replaces the search-as-home-screen pattern; does not replace the search panel itself (100% reused).

**Placement:**
- Mobile: fixed, top of `.mapwrap`, horizontally centered, `top: calc(16px + env(safe-area-inset-top))` — mirrors the existing safe-area convention already used for `.toast` (`top:calc(16px + env(safe-area-inset-top))`, line 393).
- Desktop (`min-width:821px`, post-rewrite): fixed/absolute, top-left of the now-fullscreen map, `top:24px; left:24px` — does NOT centre on desktop (centering competes visually with map controls/Leaflet zoom control which Leaflet places top-left by default; offsetting right of the Leaflet zoom control, or top-center, is Claude's discretion at implementation — either is compliant with this contract as long as it does not overlap `.leaflet-control-zoom`).
- z-index: above the map (`#map` has no explicit z-index; Leaflet panes are internally stacked), below `.detail` (z-index:500) and below `.loading`/`.calc`/`.wiz` (z-index:850/2000) — pill z-index: **400**.

**Sizing:**
- Height: min 44px (touch target compliance, matches `.laudobtn`, `.cmpbtn`, `.moderow button` convention of `min-height:44px`).
- Width: intrinsic (hug content: icon + label + padding), NOT full-bleed — this is a pill, not a bar, to visually signal "floating card over map" rather than "docked toolbar." Max-width on mobile: `calc(100vw - 32px)` so it never touches screen edges.
- Padding: 12px vertical / 16px horizontal (rounds to the 16px `md` token horizontally, slightly under on vertical to hit the 44px height target with a 15px label).

**Shape/surface:**
- `background: var(--paper-2)`
- `border: 1.5px solid var(--ink)` (matches `.wcard{border:1.5px solid var(--line)}` weight convention, but uses `--ink` not `--line` so the pill has enough contrast to read clearly floating over variable map tile colors — a `--line` border would be too faint over light basemap tiles)
- `border-radius: 999px` (true pill shape — this is a new radius value for the app, which otherwise uses `border-radius:2px` for rectangular chrome; a pill is categorically a different affordance shape signaling "floating, tappable, map-context control," similar to how `.seg button{border-radius:22px}` and `.cseg button{border-radius:10px}` already break from the 2px convention for rounder, friendlier controls)
- `box-shadow: 0 4px 14px rgba(20,26,31,.18)` (slightly stronger than the app's flat `--shadow` token since the pill floats over imagery, not over the flat paper background the `--shadow` token was designed for; consistent in spirit with `.toast{box-shadow:0 4px 14px rgba(0,0,0,.25)}`)

**Content:**
- Icon: 🔍 (reuse of existing `#vbBusca` glyph)
- Label: "O que você procura?" (locked in CONTEXT.md)
- Layout: icon + label, `gap:8px` (`sm` token), icon left of label

**States:**
| State | Visual |
|-------|--------|
| Default/idle | `background:var(--paper-2); border:1.5px solid var(--ink); color:var(--ink)` |
| Hover (desktop only) | `border-color:var(--accent-ink)` (subtle, no background change — matches `.modemore:hover{color:var(--accent)}` restraint) |
| Active/pressed (both) | `border-color:var(--accent)` + existing app-wide `:active{transform:scale(.98)}` convention if already present elsewhere (confirm against IDEIAS doc reference in FEATURES.md — if not yet in v1.0 CSS, this is a no-op reference, not a requirement to add now; motion is Phase 6) |
| Focus-visible (keyboard) | `outline:2px solid var(--accent); outline-offset:2px` — reuses the exact global rule at line 88, since the pill is a real `<button>` and inherits it automatically if selector list is extended, or explicitly if not |
| Disabled | not applicable — pill has no disabled state, always tappable |

**Accessibility:**
- Element: real `<button>`, not a styled `<div>`.
- `aria-label="Abrir busca de imóvel"` (explicit, since the visible label + icon combination could be ambiguous to screen readers about what "opening" means — the pill visually implies a text input but is actually a button that reveals one).
- `aria-haspopup="dialog"` is NOT accurate if the destination is `setView('busca')` on mobile (a view swap, not a dialog) — use `aria-haspopup="dialog"` ONLY for the desktop overlay variant (see Layout section below). On mobile, omit `aria-haspopup` since it's a view transition, not a popup.
- Keyboard: pill is a normal tab-stop; `Enter`/`Space` activates it (native button behavior, no custom key handling needed).
- On activation: focus must move into the revealed search panel's first focusable field. Mobile: reuse `focusFirstField()` (already exists, line 857) after `setView('busca')`. Desktop overlay: focus the first field of `.panel` markup on overlay open (new call site, same function).
- Esc: on desktop overlay only, `Esc` closes the overlay and returns focus to the pill (standard overlay-dismissal pattern, matches the wizard's existing modal-return-focus discipline referenced in CONTEXT.md `specifics`). On mobile, `Esc` is not applicable (no hardware Esc on touch; the existing `#vbMapa` button and/or a close affordance in the search view handles return-to-map).

### 2. Map-First Layout — Mobile

- No structural change beyond the boot default: `body[data-view]` initial value flips from `"busca"` to `"mapa"`.
- `#vbBusca` loses its initial `on` class; `#vbMapa` gains it (or `setView('mapa')` runs once on boot after `initMap()` — Claude's discretion on mechanism, contract requirement is only the resulting state: map visible, search hidden, `#vbMapa` shows the `.on`/accent-filled state per existing `.viewbar button.on{background:var(--accent);color:#fff}`).
- Existing `@media(max-width:820px)` block (lines 382-413) is otherwise UNCHANGED — `.viewbar` (Busca⇄Mapa bottom bar) remains, per CONTEXT.md decision ("a barra inferior existente pode permanecer como reforço no mobile"). The pill is additive, layered on top of the map view, not a replacement for the bottom bar.
- `map.invalidateSize()` must still fire whenever the view becomes `mapa` (already wired via `setView()`, line 874) — since map now boots directly into visibility, confirm `initMap()` → `setView('mapa')` ordering calls `invalidateSize()` at least once after the map container has its final on-screen dimensions (i.e., after any coach-mark banner reflow, if the banner is laid out via normal flow rather than `position:fixed`/`absolute` — see Coach-mark section, which specifies `position:absolute` overlay to avoid this exact reflow risk).

### 3. Map-First Layout — Desktop (rewrite of `@media(min-width:821px)` region)

**Current state (line 62, unguarded default):** `.wrap{display:grid;grid-template-columns:390px 1fr}` — panel and mapwrap permanently side-by-side, `.viewbar{display:none}`.

**New contract:**
- Add a `@media(min-width:821px)` block that overrides the base grid: `.wrap{display:block}` (or `grid-template-columns:1fr`), so `.mapwrap` occupies the full viewport width and height on desktop too.
- `.panel` on desktop becomes an **overlay card**, hidden by default, shown only when the pill is activated. Reuses the EXACT SAME `.panel` markup/children (brand header, search form, results list) — no duplication, no new DOM.
- Overlay card visual treatment on desktop:
  - `position:fixed` (or absolute within `.wrap`), anchored near the pill (`top:24px;left:24px` if pill sits there, or centered — Claude's discretion, must not require scrolling to reach the close action)
  - `max-width:400px` (slightly wider than the panel's current fixed `390px` to breathe as a floating card rather than a docked rail)
  - `max-height:calc(100vh - 48px)` with internal scroll (reuses `.panel{overflow-y:auto}` already present)
  - `border:2px solid var(--ink)` (reuses `.panel{border-right:2px solid var(--ink)}` weight, now as a full border since it's a floating card not a docked rail)
  - `border-radius:4px` (new to `.panel` context but consistent with the app's other rounded-rect chrome like `.detail{border-radius:3px}`)
  - `box-shadow:0 6px 20px rgba(20,26,31,.22)` (directly reuses `.detail`'s existing shadow value verbatim, line 162 — same "floating card over map" idiom already established by the mobile bottom-sheet)
  - `z-index:450` (above pill's 400, below `.detail`'s 500, consistent stacking order — the search overlay and the lot-detail sheet should never need to coexist, but if they did, detail wins)
- The pill toggles this overlay open/closed on desktop. Suggested mechanism: reuse `setView('busca'|'mapa')` — since `data-view` currently only branches inside `@media(max-width:820px)`, extend the desktop media query to ALSO read `body[data-view]`: `body[data-view="busca"] .panel{display:block}` / default `.panel{display:none}` on desktop. This keeps ONE state model (`data-view` + `setView()`) driving both breakpoints, per CONTEXT.md's explicit instruction to "reuse `data-view`+`setView()`, not introduce a new state system."
- Closing the overlay: reuse `#vbMapa`-equivalent affordance — since `.viewbar` is mobile-only (`display:none` by default), desktop needs its own close control on the overlay card itself. Add a close "×" to `.panel` in the `.brand` header area for desktop only (44×44px, same idiom as `.detail .x`), OR rely on Esc + click-outside-the-card (clicking the visible map area behind the overlay closes it). Both are acceptable; Esc-to-close is REQUIRED regardless of which visible affordance is also present (accessibility floor, not optional).
- After a search completes with a result on desktop: currently `enquadra()` only calls `setView("mapa")` when `isMobile()` (line 809). Since desktop is now map-first too, this condition should extend to close the overlay on desktop as well post-result, so the found lot's map pin/detail sheet is immediately visible without a manual dismiss — this mirrors the mobile auto-behavior the CONTEXT.md decisions explicitly say to "preserve/extend."

### 4. Coach-mark Banner

**Purpose:** One-line, dismissible, first-use-only hint over the map. Not a modal, not blocking.

**Placement:** `position:absolute` (or fixed) inside `.mapwrap`, top of the map, BELOW the pill in visual stacking if they'd overlap — recommended: coach-mark sits directly under the pill (`top` offset = pill's top + pill height + 8px gap), full-width minus the same side margins as the pill, OR anchored to the bottom of the map just above `.viewbar`/safe-area on mobile if that reads cleaner. Exact vertical position is Claude's discretion; the constraint is: **must never overlap the pill or block map-tap targets a first-time user would want to try** (the coach-mark's own copy invites tapping a bairro, so it cannot cover the bairros it's pointing at).

**Sizing:** height auto (content-driven), min-height 44px is NOT required here since the banner itself isn't the primary tap target — only its dismiss "×" needs the 44×44px hit area.

**Surface:**
- `background:var(--paper-2)`
- `border:1px solid var(--line)` (lighter weight than the pill's `--ink` border — the coach-mark is secondary/ambient, the pill is primary/actionable; this visual hierarchy distinction is intentional)
- `border-radius:2px` (standard app rectangular convention, NOT pill-shaped — reserves the pill shape exclusively for the search affordance so users don't confuse the two floating elements)
- `box-shadow:var(--shadow)` (the app's standard flat elevation token — coach-mark is calmer/flatter than the pill, reinforcing it's dismissible chrome, not an action)
- Padding: 12px 16px (between `sm` and `md` tokens, matches `.cmp{padding:11px 12px}` density)

**Content:**
- Copy (1 line, locked structure per CONTEXT.md example): **"Toque num bairro pra explorar, ou use a busca."** — exact copy is Claude's discretion within CONTEXT.md's suggested wording; must stay ONE line at the declared 13px/1.4, must not wrap to two lines on a 375px-wide mobile viewport (test constraint, not just guidance).
- Dismiss control: "×" button, right-aligned, 44×44px hit area (glyph itself 18px per Typography section).
- No icon needed beyond the dismiss glyph — this is not a search affordance, it doesn't need the 🔍 icon.

**Behavior:**
- Shown only when `localStorage.getItem('radar_coach')` (or similarly named flag, following the exact naming register of `radar_prof`) is absent/falsy.
- Dismissing (tap "×" OR first successful map interaction — CONTEXT.md says "some após a 1ª interação" — implement BOTH triggers: explicit dismiss AND implicit dismiss-on-first-bairro-tap/search) sets the localStorage flag and removes the banner from the DOM/hides it for all future sessions.
- No animation on dismiss required for this phase (Phase 6 owns transition polish) — instant hide/removal is acceptable now.

**Accessibility:**
- `role="status"` or plain `<div>` with the dismiss button as a real `<button aria-label="Dispensar dica">×</button>`.
- Not `aria-live` assertive — it's not an urgent announcement, it's ambient first-use guidance. If it needs to be announced at all on mount, `aria-live="polite"` is sufficient, but since it's visually present at load (not injected async), a live-region announcement is likely unnecessary — Claude's discretion, default to NO aria-live to avoid a jarring first-load screen-reader interruption.
- Must be keyboard-reachable and dismissible via keyboard (Tab to the "×", Enter/Space to dismiss) — not solely a touch/mouse target.

## Interaction & Animation Notes (descriptive only — NOT implemented this phase)

Per phase boundary, Motion is Phase 6. The following are forward-looking notes for that phase, not requirements here:

- Pill → search panel/overlay reveal should eventually get a slide/fade transition (mobile: view-swap crossfade; desktop: overlay scale/fade-in from the pill's position).
- Coach-mark dismiss should eventually fade out rather than hard-hide.
- Desktop overlay open/close should eventually match whatever crossfade timing Phase 6 establishes for `.detail`/other floating cards, for visual family consistency.
- This phase implements all of the above with **instant show/hide (`display:none`/`display:block` or `.show` class toggle)**, consistent with how `.detail.show`, `.toast.show`, `.loading.show` already work with zero transition today. Do not add `transition:` CSS in this phase — that is explicitly Phase 6's job, and adding partial motion now risks inconsistent easing/duration values that Phase 6 would need to rip out.
