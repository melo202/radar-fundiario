## Color

No new hex values. Reuse existing `:root` tokens only.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--paper)` `#e9e4d8` | Map background before tiles load / non-map chrome |
| Secondary (30%) | `var(--paper-2)` `#f2eee4` | Toggle button resting surface (mirrors `.searchpill`/`#breadcrumb` chrome) |
| Accent (10%) | `var(--accent)` `#b5451f` | Toggle button `aria-pressed="true"` (satellite active) state only — border/icon color flips to accent; focus ring |
| Ink (borders/text) | `var(--ink)` `#141a1f` | Button border (resting), icon color (resting) |
| Muted | `var(--muted)` `#57503f` | Button label text (if shown) |

Accent reserved for: the satellite toggle's **active/pressed state only** (border + glyph color when `aria-pressed="true"`). Do not use accent for the resting (streets-active) state — resting state stays ink/paper-2, matching how `.caixabtn.on` flips to a saturated fill only when "on" and `.searchpill`/`#breadcrumb` stay neutral otherwise.

**Overlay legibility exception (satellite active only):** bairro (`--line` / `--accent` hover) and lote (`--lot` `#2c5545`) outline colors do NOT change hex when satellite is active — only their `opacity`/`weight` increase for contrast against dark imagery:

| Layer | Streets mode (current) | Satellite mode (adjusted) |
|-------|------------------------|----------------------------|
| `BAI_STYLE` (bairro outline) | `opacity:.8, weight:1, fillOpacity:.03` | `opacity:1, weight:1.5, fillOpacity:0` (drop fill tint entirely — imagery already has visual texture, a tan wash over satellite reads muddy) |
| `BAI_HOVER` | `opacity:1, weight:2.5, fillOpacity:.08` | unchanged — accent hover already reads at full opacity over any basemap |
| `LOT_STYLE` (lote outline) | `weight:1, fillOpacity:.05` | `weight:1.5, fillOpacity:0` (same rationale — drop the green fill wash, keep the stroke crisper) |
| `LOT_HOVER` | `weight:2.5, fillOpacity:.14` | unchanged |

Implementation note: this is a **style-function branch**, not new constants — extend `baiStyle(feature)` (already a function, not a raw object, per the codebase's own v2.1-readiness comment at line 700) and add an equivalent `lotStyle()` wrapper around the current `LOT_STYLE` object, both reading a single module-level `satelliteOn` boolean. No new hex enters the palette.

Destructive color: not applicable — this phase has no destructive actions.

## Component Contract: Satellite Toggle Button

### Placement

Map corner, avoiding collision with existing overlays:

- **Desktop** (`.mapwrap{position:absolute;inset:0}` breakpoint, ≥ the existing desktop media query): Leaflet's default `zoomControl:true` occupies `.leaflet-top.leaflet-left` (top-left corner of the `#map` div, Leaflet default offset ~10px from map edge). The `.searchpill`/`.panel` trigger and `#breadcrumb` also anchor top-left (`top:24px;left:24px`) as page-level absolutely-positioned overlays OUTSIDE the Leaflet control system (they sit in `.mapwrap`, not in a Leaflet pane). **Place the satellite toggle bottom-right of the map**, as a native `L.control` in Leaflet's `bottomright` position slot. This is the one corner with zero existing occupants in both mobile and desktop layouts (checked: `.detail` sheet uses `left:16px;right:16px;bottom:16px` full-width on mobile, but only shows when a lote is selected — see collision note below).
- **Mobile**: same `bottomright` Leaflet control slot. **Collision note:** the `.detail` bottom sheet (`position:absolute;left:16px;right:16px;bottom:16px`) becomes full-width and would sit UNDER a bottom-right map control when open (`.detail` z-index 500 > Leaflet control default z-index ~1000... actually Leaflet controls default to a HIGHER z-index than `.detail`'s 500). **Resolve by setting the toggle control's z-index explicitly to 490** (below `.detail`'s 500, above the map's tile/vector panes) so that when a lote card sheet is open, the sheet visually covers the toggle rather than floating on top of it. This matches the existing z-index ladder: tiles/panes (370/380) < overlayPane (400) < pill/breadcrumb (400) < toggle (490, new) < `.detail` sheet (500).

### Sizing

- Button: exactly `44px × 44px` (icon-only square — no label on mobile, matches `.coach-x`/`#breadcrumb .crumb` touch-target convention).
- Desktop MAY add a text label beside the icon inside the same 44px-tall control if screen real estate allows (optional, at executor's discretion) — but the tappable icon zone itself never shrinks below 44×44px.
- Margin from map edge: `10px` (Leaflet's own default control margin — do not override, keep consistent with the existing zoom control's spacing convention already visible in the app).

### Visual identity (reuse existing toggle pattern)

Modeled directly on the existing `#btnCaixa` (`.caixabtn`) and `setMode`/`setView` `aria-pressed` button family — same border/fill logic, adapted to Leaflet-control chrome:

```
resting (streets active):
  background: var(--paper-2)
  border: 1.5px solid var(--ink)
  color: var(--ink)
  border-radius: 4px  (matches .panel/.detail radius family, NOT the pill 999px —
                        this is a squarish icon control, not a pill/label control)
  box-shadow: 0 4px 14px rgba(20,26,31,.18)  (same shadow recipe as .searchpill/#breadcrumb)

pressed (satellite active), aria-pressed="true":
  background: var(--paper-2)  (unchanged — do not invert to solid accent fill;
                                 accent is reserved for border+icon only, see Color section)
  border: 1.5px solid var(--accent)
  color: var(--accent)

hover (any state):
  border-color: var(--accent-ink)

focus-visible:
  outline: 2px solid var(--accent); outline-offset: 2px
  (identical to the app-wide focus rule at line 88 — reuse verbatim, do not redeclare)
```

### States

| State | `aria-pressed` | Border/icon color | Icon shown | Label (desktop optional) |
|-------|----------------|--------------------|------------|---------------------------|
| Streets active (default) | `"false"` | `var(--ink)` | 🛰️ | "Satélite" |
| Satellite active | `"true"` | `var(--accent)` | 🗺️ | "Ruas" |

Button element: `<button>` (not `<a>` or `<div>`), inside a custom `L.Control` (`L.control.layers` is NOT used — CONTEXT.md and codebase both point to a bespoke button, not Leaflet's built-in layers control UI, to stay on-brand). Follows the exact `aria-pressed` toggle mechanics already proven at `toggleCaixa()` (line 1885-1888) and `setMode`/`setView` (lines 976-979, 1001-1004) — one boolean state var, one function that flips `classList.toggle("on", …)` + `setAttribute("aria-pressed", …)`.

### Persistence

`localStorage` key: `radar_sat` (boolean string `"1"`/`"0"` or `"true"`/`"false"` — follow the exact serialization style already used by `radar_prof`/`radar_coach`, check those two keys' actual stored format in-code before implementing to stay consistent). Read on `initMap()` boot to restore prior session's choice; write on every toggle. This is Claude's-discretion-confirmed in CONTEXT.md — included here as the concrete key name so planner/executor don't re-decide it.

## Layer Stacking Contract

Bottom to top (z-index / Leaflet pane order), satellite mode active:

1. **Base tile pane** (`tilePane`, Leaflet default, always exactly one active base layer): Esri World Imagery `L.tileLayer` — replaces (not stacks under) the CARTO light tile. Both are added to a single `L.layerGroup` or swapped directly on the map; only one base raster layer exists at a time, per CONTEXT.md's explicit rejection of the Mapbox-GL-style "keep everything mounted" approach (this app already keeps GeoJSON overlay panes mounted — the SWAP is scoped to the raster base tile only, which is the FEATURES.md-endorsed "safer path already available in this stack").
2. **Esri Reference overlay** (`World_Boundaries_and_Places`): new Leaflet pane, `zIndex` between the base tile (`tilePane` ~200) and the existing `"bairros"` pane (370) — **set this pane's `zIndex` to 350**. Transparent tiles, visible ONLY when satellite is active (added/removed together with the satellite base tile — never independently, per CONTEXT.md's "nunca satélite pelado" rule).
3. **`"bairros"` pane** (370) — unchanged position, style-function adjusted per Color section above when satellite is active.
4. **`"lots"` pane** (380) — unchanged position, style-function adjusted per Color section above when satellite is active.
5. **`overlayPane`** (400, Leaflet default) — result pins, Caixa markers — unchanged, always on top of vector panes regardless of base mode.

This ordering guarantees: satellite imagery never occludes the reference labels, reference labels never occlude bairro/lote outlines, and outlines never occlude result pins — matching the existing pane discipline already documented in the codebase's own inline comments (lines 674, 693-694, 707).

## Crossfade Contract (SAT-02)

**Scope:** strictly the satellite base-tile swap. Does NOT apply to bairro/lote outline style changes (those snap instantly — a fill-opacity/weight change on vector strokes is cheap and instant, no transition needed) and does NOT apply to any other app transition (sheet, cards, view-switch) — those belong to Phase 6.

**Mechanism:** two-layer opacity crossfade, not a hard tile swap:

1. On toggle to satellite: add the Esri tile layer (+ reference overlay pane) at `opacity: 0`, force a reflow/next-frame, then animate to `opacity: 1` over **250ms** using `opacity` CSS transition on the tile layer's container (`.leaflet-tile-pane` or the specific tile-layer's `_container`, whichever Leaflet 1.9.4 exposes cleanly — `L.tileLayer` supports a native `opacity` option that Leaflet already animates via CSS transition on `.leaflet-tile`, so prefer calling `layer.setOpacity()` across a `requestAnimationFrame` ramp OR simply set `className` with a CSS `transition: opacity .25s ease` rule and toggle a class — either approach is acceptable, executor's choice). The CARTO base tile is removed only AFTER the satellite fade-in completes (or faded out simultaneously at the same 250ms if a true crossfade — i.e., both layers present briefly — is simpler to implement than a delayed removal; either produces the "no hard cut" perceptual result CONTEXT.md asks for).
2. On toggle back to streets: same 250ms fade, reversed.
3. Duration: **250ms** (midpoint of CONTEXT.md's stated 200-300ms range). Easing: `ease` (CSS default) — do not use a custom cubic-bezier; this is a small utility transition, not an expressive motion moment.
4. The reference overlay pane fades in/out in lockstep with the satellite base tile (same duration, same trigger) — they are visually one unit per the "never satellite without labels" rule.

**Explicitly out of scope for this phase:** any transition on `.detail` sheet open/close, `.panel` view-switch, card list stagger, or flyTo easing tuning — all deferred to Phase 6's app-wide Motion pass. Do not generalize this 250ms value into a shared CSS transition token during this phase; that generalization is Phase 6's job.

## Attribution Contract

- Rendered via Leaflet's built-in `.leaflet-control-attribution` (bottom-right of map, Leaflet default position/styling — do not restyle it, it already inherits the map's default Leaflet chrome and this phase does not touch that).
- Each `L.tileLayer` declares its own `attribution` string at creation time:
  - CARTO layer: existing string, unchanged — `'© OpenStreetMap · © CARTO'`.
  - Esri satellite layer: `'Tiles © Esri — Source: Esri, Maxar, Earthstar Geographics, and the GIS User Community'` (verbatim from CONTEXT.md).
  - Esri reference overlay layer: no separate attribution needed if it shares the same Esri/ArcGIS Online terms as the imagery layer — confirm at implementation time whether ArcGIS Online's ToS wants the reference tile service credited separately; if uncertain, append `and Esri Reference Layer contributors` is NOT required per CONTEXT.md's exact quoted string — use the CONTEXT.md string as-is and do not embellish.
- Because Leaflet's attribution control automatically unions the `attribution` strings of all currently-`addTo(map)` layers, no manual swap/replace logic is needed in JS — removing the CARTO layer and adding the Esri layer (per the crossfade sequence above) causes the attribution control to update itself automatically. This is a zero-custom-code outcome; flag it to the executor so no one hand-rolls a string-replace function.

### CSP dependency (blocking, not cosmetic — flag to planner)

The current CSP meta tag (`radar-goiania.html` line 7) restricts `img-src` to `'self' data: blob: https://*.basemaps.cartocdn.com`. **Esri tiles from `server.arcgisonline.com` will be silently blocked by this CSP** unless `img-src` is extended to include `https://server.arcgisonline.com`. This is not a design-contract concern but IS a hard prerequisite for the satellite layer to render at all — the planner must add a task for this CSP edit alongside the `sw.js` version bump already scoped in CONTEXT.md. Flagging here so it isn't lost between UI-SPEC and implementation.

## Label Legibility Contract (SAT-01)

The Esri Reference overlay (`World_Boundaries_and_Places`) IS the legibility mechanism — it ships pre-rendered place/street names with built-in halos (white/light stroke around dark text, baked into the Esri tile images themselves). This satisfies SAT-01's "no mesmo passo" requirement: there is no separate halo/CSS treatment for the app's own bairro-name tooltips to build in this phase, because:

- The app's bairro hover-tooltip labels (from Phase 3, MAPA-02) are a SEPARATE UI element from the Esri reference tiles — they are Leaflet `bindTooltip`/divIcon labels drawn by the app's own code, not baked into any tile image.
- **Scope decision:** this phase does NOT add a halo/backdrop treatment to the app's own hover-tooltip bairro labels for satellite mode. FEATURES.md's differentiator note ("Label-over-satellite legibility treatment... halo on bairro names when satellite is active") describes the Esri reference overlay's baked-in halos as satisfying this need at the base-map level. If a future audit finds the app's own hover tooltips (not the Esri reference labels) are ALSO hard to read over imagery, that is a new finding for a later phase — not assumed here, since CONTEXT.md scopes this phase to "overlay de referência Esri" only, and does not mention modifying the tooltip CSS.
- Executor check during verification: confirm Esri reference tile street/bairro names remain legible at the zoom levels this app actually uses (city zoom ~12 through lote-drill zoom ~17+) — if Esri's reference layer supplies sparse or missing labels at any zoom band the app relies on, that's a finding to report back, not a UI-SPEC gap to fix by inventing new CSS.
