## Warnings

### WR-01: `numeroPorExtenso` silently renders negative values as positive (no sign, no honesty flag)

**File:** `radar-goiania.html:159` (also affects `extensoInteiro` callers indirectly)
**Issue:** `numeroPorExtenso` does `const n=Math.abs(Number(valor));` — any negative `valorOfertado`/
`precoTotal` (e.g. a fat-fingered `-500000` from `parseInt` on user input, or a future caller
passing a signed number) is silently converted to its absolute value and printed as
`"R$ 500.000,00 (quinhentos mil reais)"` with zero indication the original value was negative.
This directly touches `propostaTexto`/`contratoTexto`, which interpolate this string into a legal
document's price clause (Cláusula 2ª/3ª). The project's own documented contract for this feature
(RADAR_PURE comments, `CAMPO_VAZIO` contract) is "nunca inventa campo ausente" / never silently
misrepresent a value — a sign flip is a stronger violation than an absent field, since the printed
document now states a materially different (wrong) price with total confidence.

In practice `NEG.valorOfertado`/`NEG.precoTotal` are populated via
`parseInt(this.value.replace(/\D/g,''),10)||null"` in `negRender()` (see `radar-goiania.html:440`,
`:487`), which strips non-digit characters and therefore cannot itself produce a negative number
today — this makes the bug currently unreachable through the wizard UI. It is, however, reachable
by any other caller of the exported `numeroPorExtenso`/`propostaTexto`/`contratoTexto` functions
(they're plain functions on `RADAR_PURE`, not wizard-only), and is exercised by no fixture/test
(`extensoCasos` has no negative case), so a future caller or refactor could reintroduce it without
any test catching the regression.

**Fix:** Either (a) reject/pass through negatives honestly instead of `Math.abs`-ing them, e.g.:
```js
function numeroPorExtenso(valor){
  if(valor==null||isNaN(valor)) return "";
  const num=Number(valor);
  const negativo=num<0;
  const n=Math.abs(num);
  const inteiro=Math.floor(n);
  const valorFmt="R$ "+n.toLocaleString("pt-BR",{minimumFractionDigits:2,maximumFractionDigits:2});
  const ext=extensoInteiro(inteiro);
  const somenteMilhoes=inteiro>=1000000 && (inteiro%1000000===0);
  const reaisTxt=somenteMilhoes ? "de reais" : "reais";
  return (negativo?"-":"")+valorFmt+" ("+(negativo?"menos ":"")+ext+" "+reaisTxt+")";
}
```
or (b) if negative values are never a legitimate business case for this app, add an explicit guard
that returns `""` (same honest-empty contract as `null`/`NaN`) so a bad value never silently prints
a wrong number:
```js
if(valor==null||isNaN(valor)||Number(valor)<0) return "";
```
Add a fixture (`extensoCasos.negativo: { valor: -500, expectEmpty: true }` or
`expectContains: ["-"]` depending on chosen fix) to lock in the behavior either way.

### WR-02: Free-text minuta editing can silently drop a whole clause from the printed PDF if it contains the disclaimer substring

**File:** `radar-goiania.html:3555` (`montarNeg`)
**Issue:** `montarNeg()` splits the (possibly user-edited) minuta on blank lines and drops any
block whose text contains the substring `"Minuta para conferência das partes"`, on the assumption
that only the literal disclaimer paragraph will ever contain it:
```js
.filter(b=>b.indexOf("Minuta para conferência das partes")<0)  // disclaimer sai do fluxo — vai pra caixa
```
The wizard explicitly invites free-text editing ("O texto abaixo já está pronto — edite qualquer
trecho antes de gerar o PDF", `radar-goiania.html:510`). If a user's edit to any clause
legitimately includes that phrase — e.g. adding a note like *"As partes reconhecem que esta Minuta
para conferência das partes já foi revisada por advogado em 01/03"* inside Cláusula 4ª — the
**entire clause block silently disappears** from the printed A4 document with no warning, no log,
no indication to the user that content was dropped. This is a real content-loss bug in a
legal-document generator: the disclaimer box (`.negdisclaimer-print`) is still (correctly, and
separately) always injected from the `DISCLAIMER_NEG` constant, so this bug affects only the
*user's own edited clause text*, not the disclaimer requirement — but a party relying on the
printed contract they believe matches what they reviewed on screen will get a document missing a
clause they wrote.

Verified via direct simulation:
```
Input blocks: 4 (title, Cláusula 1ª, Cláusula 4ª containing the substring, Cláusula 5ª)
Output blocks kept: 3 — Cláusula 4ª silently vanished.
```

**Fix:** Match the disclaimer block by structural equality (or startsWith after trim), not
substring containment, so only the actual disclaimer paragraph is filtered:
```js
.filter(b=>b.trim()!==DISCLAIMER_NEG.trim())
```
Since `blocosBrutos` entries are already `.trim()`-ed, and the disclaimer is always appended by the
pure functions as its own isolated `\n\n`-delimited block (`texto+="\n"+DISCLAIMER_NEG+"\n";`),
exact-match is safe and will only ever strip the real disclaimer paragraph, not a user's own prose
that happens to reference it.

### WR-03: Título heuristic in `montarNeg` misclassifies numeric/short single-line blocks as clause headings

**File:** `radar-goiania.html:3561`
**Issue:**
```js
const ehTitulo=/^Cláusula\s+\d+ª?\s*—?/.test(primeira)||primeira===primeira.toUpperCase();
```
`x === x.toUpperCase()` is trivially true for any string with no lowercase letters — including
purely numeric strings (`"500"`), currency strings (`"R$ 500.000,00"`), or an empty string. If a
user edits the minuta and, e.g., splits a paragraph so that a value or short all-caps/no-letter
line stands alone as its own blank-line-delimited block, that block is rendered as a bold
"heading" (`<b>...</b>`) with **no body paragraph at all** when it has no second line (`resto===""`
and `ehTitulo===true` → the `<p>` branch is skipped). The content isn't lost from the DOM (unlike
WR-02), but it is visually misrepresented as a clause title in the printed legal document, which is
a document-integrity/quality issue rather than a security one.

**Fix:** Require the title line to contain at least one letter before applying the
all-uppercase heuristic, e.g.:
```js
const pareceTitulo=/[A-ZÀ-Ú]/.test(primeira) && primeira===primeira.toUpperCase();
const ehTitulo=/^Cláusula\s+\d+ª?\s*—?/.test(primeira)||pareceTitulo;
```

## Info

### IN-01: Duplication across `propostaTexto`/`termoExclusividadeTexto`/`contratoTexto`

**File:** `radar-goiania.html:218-351`
**Issue:** The three minuta functions repeat the same structural pattern (title line, party
block via `nomeOuVazio`+`docEnderecoTxt`, numbered `Cláusula Nª — ...\n` string concatenation,
trailing `DISCLAIMER_NEG` + two `blocoAssinatura` calls). This is already partially factored
(`nomeOuVazio`, `docEnderecoTxt`, `imovelDescTxt`, `blocoAssinatura` are shared), which is good,
but the "Cláusula Nª — Do Foro" clause text and the `"\n"+DISCLAIMER_NEG+"\n"` trailer are
duplicated verbatim three times.
**Fix:** Low priority — a `clausulaForo(foro, n)` helper and a `rodapeDisclaimerEAssinaturas(...)`
helper would remove the last bit of duplication, but given each function has genuinely different
clause counts/conditionals (the exclusividade conditional 5ª clause in particular), the current
level of duplication is reasonable and doesn't currently risk drift (all three literal foro
clauses are character-identical today, but nothing enforces that as they evolve independently).

### IN-02: No `aria-hidden` management when `#negSheet` opens on top of `#captSheet`

**File:** `radar-goiania.html:912`, `radar-goiania.html:3347` (`abrirNeg`)
**Issue:** When `abrirNeg('exclusividade')` is invoked from the button inside `#captSheet`
(`radar-goiania.html:904`), `#captSheet` is never hidden — it remains `hidden=false` underneath
`#negSheet`, relying purely on visual z-order (both share `.wiz{z-index:2000}`, DOM order breaks
the tie). Screen-reader users could in theory still navigate into the backgrounded `#captSheet`'s
focusable elements while `#negSheet` is the active modal. This is a pre-existing gap in the
`#wiz`/`#captSheet` nesting pattern (not introduced by 11.1), now inherited by the new
`#negSheet`-inside-`#captSheet` case.
**Fix:** Out of scope for this phase (systemic, affects all three sheets equally) — worth a
follow-up ticket to add `aria-hidden="true"` toggling on background sheets when a nested sheet
opens, applied consistently across `#wiz`/`#captSheet`/`#negSheet`.

### IN-03: `DISCLAIMER_NEG` string duplicated literally in the step-1 review UI markup

**File:** `radar-goiania.html:514`
**Issue:** The step-1 review screen hardcodes the disclaimer text a second time as a literal
string inside the `negRender()` template:
```js
<div class="negdisclaimer-txt">Minuta para conferência das partes — recomenda-se revisão por advogado; ...</div>
```
instead of interpolating the existing `DISCLAIMER_NEG` constant (which is used correctly via
`${esc(DISCLAIMER_NEG)}` in `montarNeg()`'s print box). Today the two strings are character-identical,
but nothing enforces that — a future edit to one and not the other would make the on-screen
review text diverge from the printed disclaimer, which is exactly the kind of drift the constant
was introduced to prevent (per the `DISCLAIMER_NEG`/`CAMPO_VAZIO` "single source of truth" comment
at `radar-goiania.html:91`).
**Fix:**
```js
<div class="negdisclaimer-txt">${esc(DISCLAIMER_NEG)}</div>
```
