# Phase 3 — UI Design Contract

> Visual and interaction contract for: Render de Bairro (polígonos-linha) + Hover/Tap + Click-to-Drill + Breadcrumb.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Covers MAPA-02, MAPA-03, MAPA-05. Reuses the app's existing cartographic identity — zero new hex, zero new component library.

Declared values (must be multiples of 4) — reuses the app's existing informal scale (no formal token names in the codebase; the app uses ad-hoc px values that already land on 4/8-multiples in >90% of rules):

| Token | Value | Usage in this phase |
|-------|-------|-------|

- Bairro polygon stroke `weight` values (1 / 2.5) are **not spacing** — see Color/Style section; these mirror `LOT_STYLE`/`LOT_HOVER` weights exactly (1 and 2.5), not arbitrary.

## Color

Existing 60/30/10 split (unchanged app-wide; this phase does not alter global color proportions, only extends usage of existing tokens to two new elements):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--paper)` `#e9e4d8` | Map background (`#map` already uses this); unchanged |
| Secondary (30%) | `var(--paper-2)` `#f2eee4` | Breadcrumb pill background (mirrors `.searchpill` and `.coachmark` backgrounds, both `var(--paper-2)`) |
| Accent (10%) | `var(--accent)` `#b5451f` | **Reserved for, in this phase, ONLY:** (1) bairro polygon stroke on hover/tap highlight (`BAI_HOVER.color`), (2) the clickable "Goiânia" crumb label color, (3) breadcrumb separator hover is NOT accented (separator is never interactive) |
| Destructive | not applicable | No destructive actions in this phase (no delete/remove UI introduced) |

**Bairro layer color contract (zero new hex — every value below is an existing `var()`):**

```
BAI_STYLE (idle, zoom < gate threshold):
  pane: "bairros"  (new pane, see Z-Index section)
  color: var(--line)      /* #c3b9a3 — existing "divisa" stroke color, same semantic role LOT_STYLE gives var(--lot) but for administrative boundaries, not cadastral ones — deliberately distinct from lot green so bairro ≠ lote at a glance */
  weight: 1
  fillColor: var(--line)
  fillOpacity: .03         /* near-transparent per MAPA-02 "linha", not filled choropleth */
  opacity: .8
  bubblingMouseEvents: false

BAI_HOVER (highlighted — desktop mouseover / mobile 1st tap):
  color: var(--accent)     /* #b5451f — same accent used for LOT_HOVER, keeps "you're focused on this" feedback visually consistent app-wide */
  weight: 2.5              /* identical weight delta to LOT_HOVER (1→2.5), same visual grammar */
  fillOpacity: .08          /* subtle fill lift, stays well below LOT_HOVER's .14 since bairros are much larger shapes — avoids a "painted" look at city zoom */
  opacity: 1
```

Rationale for reusing `--line` instead of `--lot` or a new color: `--lot` (`#2c5545`, dark green) is semantically "lote boundary" throughout the app (`LOT_STYLE`, `.card` left-border, `.cmpbtn`). Bairros are a different administrative tier — reusing `--line` (already the app's generic "divisa"/thin-line-on-paper token, used for dashed section dividers) keeps the visual hierarchy legible: bairro = thin neutral line, lote = green line, hover(either) = accent red. No new hex satisfies this without breaking that grammar.

**Unnamed rural gleba (`nm_bai === null`, ~466 features):** same `BAI_STYLE`/`BAI_HOVER` — visually identical polygon treatment. Only the tooltip text differs (see Copywriting). Do not invent a visually distinct "muted" style for glebas; they are still real administrative boundaries and should hover/drill identically. This avoids a silent second visual system for ~28% of features (466 of ~1206).

## Z-Index / Layout Coexistence

| Layer | z-index | Notes |
|-------|---------|-------|
| Bairro polygon pane (`map.createPane("bairros")`) | **370** | Below existing `"lots"` pane (380) and below Leaflet's `overlayPane` (400) — bairro outlines must sit visually *under* everything else on the map, including tile labels and any future pin/marker layer. Sits below lots pane so if both are ever visible in the same frame during a transition, lots win. |
| `#searchPill` | 400 | Unchanged from Phase 2. |
| `#breadcrumb` (new) | **400** | Same tier as the pill — they are siblings in the UI chrome, not stacked. They do NOT overlap spatially (see placement below), so sharing a z-index is safe and avoids inventing a new tier. |
| `.coachmark` | 390 | Unchanged. Sits below both pill and breadcrumb — coachmark text must never be read as "on top of" the new breadcrumb; if both are visible simultaneously (first-run + already-drilled edge case, unlikely but possible), verify coachmark still reads left-anchored under the pill's vertical band, not colliding with the now-left-anchored breadcrumb. **Discretion note for executor:** if overlap is visually detected in testing, coachmark should be dismissed automatically on drill (call `dismissCoach()` inside the drill handler) rather than repositioned — simpler, matches existing "coach dismisses on first meaningful map action" intent. |
| `.detail` sheet | 500 | Unchanged. |

**Placement rule to prevent pill/breadcrumb collision:** `#searchPill` is horizontally centered (`left:50%; transform:translateX(-50%)`). `#breadcrumb` is left-anchored (`left:16px`, no transform). On mobile narrow viewports (`max-width:640px`), verify the breadcrumb's max rendered width (worst case: "Goiânia › " + longest bairro name, e.g. "JARDIM GOIÁS") does not visually reach the pill's left edge. **Contract:** breadcrumb must apply `max-width: calc(50vw - 24px)` with `text-overflow:ellipsis; white-space:nowrap; overflow:hidden` on the `.crumb.bairro` span specifically (not the whole breadcrumb) so long bairro names truncate rather than push into the pill's space. This is a measurable, testable constraint — not aesthetic judgment.

On the desktop breakpoint where `.searchpill` moves to `top:24px;left:24px` (line 438, no transform, corner-anchored, offset from zoom control), the breadcrumb must render **below** the pill in that same left column, not beside it: `top: calc(24px + 44px + 8px)` i.e. stacked vertically with an 8px gap, both left-anchored at 24px. This is a deliberate breakpoint-specific override — note it explicitly for the executor since it's a different placement axis (horizontal-separation on mobile vs. vertical-stacking on desktop corner layout).

## States

### Bairro layer visibility (zoom-gated)

| Zoom range | Bairro layer | Lot layer (`refreshLots`, unchanged) |
|---|---|---|
| ≤ 16 | **visible** (`BAI_STYLE`), interactive (hover/tap enabled) | hidden (existing `<17` gate, untouched) |
| ≥ 17 | **hidden** (`map.removeLayer` or pane display:none — executor's discretion which mechanism, but must fully detach interactivity, not just visually fade, since Motion/fade is out of scope for this phase) | visible (existing `refreshLots()` behavior, untouched) |

Chosen threshold: **hide bairros at zoom ≥ 17**, matching the lot layer's existing entry threshold exactly (no gap, no overlap zoom band) — simplest mental model: "one boundary system is always in charge, never both drawn at once, the handoff point is a single number already established by `refreshLots()`." Toggle on `zoomend` (reuse the existing `moveend`→debounce pattern already wired for `refreshLots`; bairro toggle can piggyback on the same listener or a sibling one — executor's discretion, not a new architectural pattern).

### Hover/tap highlight (desktop vs. mobile, one function per CONTEXT.md decision)

| Input | State 1 (idle) | State 2 (highlighted) | State 3 (drilled) |
|---|---|---|---|
| Desktop `mouseover` | `BAI_STYLE` | `BAI_HOVER` + tooltip showing `nm_bai` (sticky, direction "top", opacity .96 — copy `LOT` tooltip binding options exactly) | N/A — desktop click drills directly, no 2-step |
| Desktop `mouseout` | returns to `BAI_STYLE`, tooltip removed | — | — |
| Desktop `click` | — | — | `fitBounds` on the bairro's geometry bounds (pad ~0.05, matching `mostrarBairro`'s existing pad convention at line 850) → breadcrumb appears |
| Mobile 1st tap (`matchMedia("(hover:none)")`) | `BAI_STYLE` | `BAI_HOVER` + tooltip showing `nm_bai` (same visual as desktop hover — tooltip must open on tap since there's no hover to reveal it any other way) | — |
| Mobile 2nd tap (same bairro, already highlighted) | — | — | `fitBounds` + breadcrumb appears (drill) |
| Mobile tap on a DIFFERENT bairro while one is highlighted | previous bairro returns to `BAI_STYLE` | new bairro becomes `BAI_HOVER` + tooltip | — |

This is the exact `highlightBairro()` single-function contract from CONTEXT.md — desktop fires it on `mouseover`, mobile fires it on `click`/`tap`, and the drill logic is gated by "already highlighted?" on mobile vs. "any click" on desktop. Named states above are for spec clarity; the implementation is one function with a touch-detection branch, matching `refreshLots()`'s own precedent.

### Breadcrumb visibility

| App state | Breadcrumb |
|---|---|
| Home (city-wide view, zoom ≤ ~13, no bairro drilled) | `display:none` |
| Drilled into a bairro (post-`fitBounds`) | `display:inline-flex`, content = `Goiânia › {NM_BAI or "Área rural"}` |
| User taps "Goiânia" crumb | `map.setView([-16.6799,-49.255],12)` (exact boot center/zoom, per CONTEXT.md) → clears bairro highlight state → breadcrumb returns to `display:none` |
| User manually zooms/pans out below the drill threshold without tapping the crumb | **Discretion, recommend:** breadcrumb persists showing the last-drilled bairro name until the crumb is explicitly tapped OR the user drills into a different bairro (whichever first) — avoids flicker from incidental zoom jitter; do NOT tie breadcrumb visibility to the same zoom gate as the lot layer, that would cause it to disappear/reappear on every zoom crossing, which reads as broken, not premium |

| Element | Copy |
|---------|------|

| Error state (GeoJSON fetch fails) | If `fetch('bairros-goiania.json')` fails or the service worker's precache is stale/missing: **fail silently, do not block the map.** The bairro layer simply doesn't render; lots and search still work. No toast, no error banner — this mirrors the CONTEXT.md principle of the bairro layer being a progressive enhancement over the core search flow, not a blocking dependency. (If executor judges a toast is warranted for debugging visibility in dev, keep it console-only, not user-facing.) |
| Destructive confirmation | Not applicable — no destructive actions introduced in this phase |

## Interaction Notes (non-visual, for executor — explicitly NOT Motion/Phase-6 scope)

- No `transition:` CSS property is introduced for the bairro layer or breadcrumb by this phase. Leaflet's `setStyle()` calls (as used by `LOT_STYLE`/`LOT_HOVER` today) apply instantly — mirror that exactly, do not add easing.
- The breadcrumb's show/hide is a hard `display` toggle (`none` ⇄ `inline-flex`), not an opacity fade — consistent with `.pillclose`/`.coach-x` show/hide patterns elsewhere in the app, which are also instant.
- `fitBounds` itself has Leaflet-default pan/zoom animation (this is existing default map behavior already present via `mostrarBairro`'s own `fitBounds` call at line 850 — not new motion introduced by this phase, just reused default behavior). Do not add `{animate:false}` (would regress existing feel) and do not add custom easing/duration options (would be scope creep into Phase 6).

## What's Explicitly OUT of This Phase

- **Satélite / tile-layer switching** — Phase 4. This spec assumes the CARTO light basemap only.
- **Motion polish** (flyTo custom easing/duration, crossfade, stagger-in for lot polygons, breadcrumb fade) — Phase 6. Any transition-like behavior described above is either (a) Leaflet's existing default, unmodified, or (b) explicitly a hard instant toggle.
- **Heatmap / territory-aware styling** (`isMyTerritory` style branch, R$/m² gradient) — v2.1+. This spec's `BAI_STYLE` is written as a flat object today but the executor should keep style resolution as a small function of feature properties (not hardcoded inline in `L.geoJSON`'s `style:` option as a bare object) so a v2.1 conditional branch is a cheap addition later — this is an architecture note, not a v2.0 visual requirement.
- **Live ArcGIS fetch for bairro polygons** — explicitly forbidden per CONTEXT.md; this phase renders only the static `bairros-goiania.json` Fase 1 artifact via `L.geoJSON()`, no `toWGS()` reprojection needed (already WGS84).
- **1206 always-on fixed labels** — anti-feature per FEATURES.md research; names appear only on highlight (hover/tap), never as a permanent label layer.
- **Blue-dot / locate-me, satellite label-legibility treatment** — later phases per FEATURES.md MVP staging.
